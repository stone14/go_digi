## 1. 왜 Digicap인가 — 7가지 차별점

### ① 엑셀 대장이 사라진다 — "자산 등록 = 운영 시작"

기존에는 장비를 도입하면 엑셀에 수동 입력하고, 별도로 모니터링 툴에 등록하고, 또 별도로 구성도를 그려야 했습니다. Digicap에서는 **자산을 한 번 등록하면 모니터링·토폴로지·랙 배치·알림이 자동으로 연결**됩니다. 에이전트를 설치하면 자산 등록조차 자동입니다. 엑셀 대장을 열 일이 없어집니다.

> **핵심**: 자산(Asset) 하나의 ID에 메트릭·랙 위치·토폴로지·계약·변경 이력·장애 이력이 전부 연결되는 **자산 중심 아키텍처**. 분산된 정보를 하나로 모으는 것이 아니라, 처음부터 하나로 생성됩니다.

### ② 구성도가 스스로 그려진다 — "그리지 않아도 되는 토폴로지"

대부분의 조직에서 구성도는 SI 업체가 납품 시 작성한 Visio 파일이 마지막입니다. 이후 장비가 추가되고, 포트가 변경되어도 구성도는 업데이트되지 않습니다.

Digicap은 에이전트가 수집하는 **LLDP·ARP·MAC·WWN 데이터를 실시간으로 분석**하여 토폴로지를 자동 생성합니다. 장비를 추가하거나 케이블을 변경하면 구성도가 자동으로 반영됩니다. 네트워크 스캔으로 에이전트가 없는 장비도 자동 발견합니다.

> **핵심**: 사람이 그리는 구성도는 만든 순간부터 노후화됩니다. 시스템이 자동으로 유지하는 구성도만이 항상 정확합니다.

### ③ 담당자가 바뀌어도 아무것도 잃지 않는다 — "시스템이 기억한다"

이것이 Digicap의 가장 강력한 차별점입니다. 모든 모니터링 툴은 현재 상태를 보여주지만, **과거의 맥락**까지 보존하는 툴은 거의 없습니다.

- **왜 이 서버의 IP가 바뀌었는지?** → 변경 이력에 기록되어 있습니다
- **3개월 전에도 같은 장애가 있었는데 어떻게 해결했는지?** → Incident 타임라인에 기록되어 있습니다
- **이 방화벽 룰은 누가 왜 추가했는지?** → Syslog 기반 설정 변경 기록에 남아 있습니다
- **이 서버의 유지보수 업체 담당자가 누구인지?** → 자산에 연결된 계약 정보에 있습니다

신규 담당자가 첫날 출근해서 대시보드를 열면, 전체 인프라 현황·구성도·장애 이력·변경 이력을 즉시 확인할 수 있습니다. **온보딩 기간이 수개월에서 수일로 단축**됩니다.

> **핵심**: 모니터링 툴은 "지금 무슨 일이 일어나고 있는가"를 보여줍니다. Digicap은 "왜 이렇게 되었는가"까지 보여줍니다. 그 차이가 담당자 교체 시 드러납니다.

### ④ 서버 1대에서 돌아간다 — 도입 장벽 제로

Zabbix + Grafana + ELK + CMDB를 조합하면 최소 서버 3~4대, 설치·연동에 수주가 소요됩니다. Digicap은 **PostgreSQL 하나와 Node.js 서버 하나**로 전체 기능이 동작합니다.

- 별도 메시지 큐(Kafka, Redis) 불필요
- 별도 시계열 DB(InfluxDB, Prometheus) 불필요
- 별도 검색엔진(Elasticsearch) 불필요
- 설치부터 운영 시작까지 **1시간 이내**

> **핵심**: 복잡도는 기능의 적이 아니라 아키텍처의 문제입니다. 31개 페이지, 50+ API, AI 분석까지 서버 1대로 돌아가는 것이 Digicap의 설계 철학입니다.

### ⑤ 폐쇄망에서도 AI가 동작한다

금융·공공·의료 환경에서 AI를 쓸 수 없는 이유는 단 하나, 데이터가 외부로 나가기 때문입니다. Digicap은 **Ollama 로컬 LLM**을 지원하여 인터넷 연결 없이도 이상 탐지·용량 예측·로그 분석·자연어 질의가 가능합니다.

인터넷이 가능한 환경에서는 OpenAI·Anthropic API로 전환하여 더 높은 분석 정확도를 얻을 수 있습니다. 설정 페이지에서 클릭 한 번으로 전환됩니다.

> **핵심**: AI가 선택이 아니라 기본입니다. 환경에 따라 로컬/클라우드를 유연하게 선택할 수 있습니다.

### ⑥ 모니터링이 아니라 "운영"을 통합한다

Zabbix는 훌륭한 모니터링 툴입니다. 하지만 자산 대장을 관리하지 않고, 장애 이력을 기록하지 않고, 유지보수 계약을 추적하지 않고, 구성도를 자동 생성하지 않습니다. 결국 모니터링 외의 업무는 여전히 엑셀·이메일·카톡에 흩어집니다.

Digicap은 모니터링 툴이 아니라 **운영 플랫폼**입니다.

| 업무 | Zabbix | Digicap |
|------|--------|---------|
| CPU·메모리 모니터링 | ✓ | ✓ |
| 장비 대장 관리 | ✗ (별도 엑셀) | ✓ |
| 유지보수 계약·만료 추적 | ✗ (별도 엑셀) | ✓ |
| IP 대역 관리 | ✗ (별도 엑셀) | ✓ |
| SSL/도메인 만료 경보 | ✗ | ✓ (매일 자동) |
| 토폴로지 자동 생성 | △ (수동 설정) | ✓ (LLDP/ARP/WWN) |
| 랙 실장 현황 | ✗ | ✓ |
| 장애 이력·타임라인 | ✗ | ✓ |
| 변경 이력 자동 기록 | △ | ✓ (Syslog 연동) |
| AI 예측 분석 | ✗ | ✓ |
| 라이프사이클 관리 (등록~폐기) | ✗ | ✓ |

> **핵심**: 모니터링은 운영의 일부입니다. Digicap은 모니터링 포함 운영 전체를 하나의 프로세스로 통합합니다.

### ⑦ 한국 인프라 운영팀을 위해 설계되었다

- **전체 한국어 UI**: 메뉴, 알림, 보고서, 에러 메시지까지 모두 한국어
- **국내 운영 환경 반영**: 유지보수 계약 구조 (벤더·담당자·만료일), 자산 라이프사이클 (운영→폐기예정→폐기완료→반납), 감사 추적 요구사항
- **SI 납품 후 운영 시나리오 고려**: SI 업체가 구축하고 떠난 뒤, 내부 담당자가 이어받는 국내 현실에 맞춘 설계
- **중소·중견 IT팀 규모 최적화**: 2~5명이 전 영역을 담당하는 현실, 전문 관리자 없이도 운영 가능한 자동화 수준

> **핵심**: 해외 NMS를 한국어로 번역한 것이 아니라, 한국 인프라 운영팀의 실제 업무 흐름에서 설계된 제품입니다.

## 2. 제품 비전

### "사람이 아닌 시스템이 기억하는 인프라 운영"

국내 중소·중견 규모의 인프라 운영팀은 대부분 2~5명이 서버·네트워크·스토리지·보안을 동시에 담당합니다.
장비 대장은 엑셀, 모니터링은 Zabbix, IP 관리는 또 다른 엑셀, 장애 이력은 카카오톡 — 업무 도구가 파편화되어 있고, 담당자가 바뀌면 모든 이력과 노하우가 사라집니다.

**Digicap**는 이 문제를 근본적으로 해결합니다.

자산 등록 → 모니터링 설정 → 토폴로지 자동 구성 → 알림 → 장애 대응 → 이력 관리 → 용량 예측까지, 인프라 운영의 전체 생애주기를 **하나의 시스템** 안에서 수행합니다. 담당자가 바뀌어도 시스템에 축적된 이력·구성도·장애 기록이 그대로 남아, 신규 담당자가 빠르게 업무를 파악하고 이어갈 수 있습니다.

### 핵심 가치

| 가치 | 설명 |
|------|------|
| **하나의 프로세스** | 자산 등록부터 폐기까지 전체 생애주기를 단일 시스템에서 관리 |
| **이력의 영속성** | 모든 변경·장애·대응 기록이 DB에 축적 — 담당자가 바뀌어도 소실 없음 |
| **자동 현행화** | 에이전트가 상태를 자동 수집하고 토폴로지가 자동 생성 — 구성도가 항상 최신 |
| **통합 가시성** | 서버·네트워크·스토리지·보안 전 계층을 단일 대시보드에서 확인 |
| **선제적 대응** | AI 기반 이상 탐지와 용량 소진 예측으로 장애 발생 전 대응 |

## 3. 기존 인프라 운영의 문제

> 국내 온프레미스 인프라를 운영하는 대부분의 기업은 **전담 인력도, 전문가도 없이** 여러 툴을 혼용하며 운영하고 있습니다.
> 담당자가 바뀔 때마다 그간의 모든 이력과 노하우가 함께 사라지는 것이 현실입니다.

### 3-1. 인력 현실 — "인프라 전담이 없다"

| 유형 | 실제 운영 방식 |
|------|--------------|
| **중소기업 (~300명)** | 개발자 또는 전산 담당자 1명이 서버·네트워크·보안·PC 전부 담당 |
| **중견기업 (300~1000명)** | IT 팀 2~3명이 인프라 전체를 나눠서 담당, 전문 영역 구분 없음 |
| **공공기관·지자체** | 순환 보직으로 2년마다 담당자 교체, 인프라 전문성 축적 불가 |
| **병원·학교** | 의료·교육 전공자가 IT를 겸직, 인프라 지식이 거의 없는 상태 |

- 인프라 전담 인력이 있더라도 **서버·네트워크·스토리지·보안 전 영역의 전문가**인 경우는 드뭄
- 장비 도입 당시 SI 업체가 구성하고 떠난 뒤, **"왜 이렇게 설정됐는지" 모르는 채로 운영**하는 경우가 대부분
- 전문 지식 부족으로 장애 발생 시 제조사·유지보수 업체에 의존 → **대응 시간 수 시간~수일**

### 3-2. 히스토리 소실 — "담당자가 바뀌면 모든 게 사라진다"

**가장 큰 문제는 히스토리가 사람에게 귀속된다는 것입니다.**

```
담당자 A 재직 기간 (3년)
  └─ 서버 구성 변경 이력   → 머릿속·개인 메모장
  └─ 장애 대응 노하우      → 개인 경험
  └─ 네트워크 구성도       → 개인 다이어그램 파일
  └─ 장비 비밀번호         → 개인 메모
  └─ 유지보수 업체 연락처  → 개인 주소록

담당자 A 퇴사 →  위의 모든 정보 전부 소실
담당자 B 입사 →  처음부터 다시 파악 (수개월 소요)
```

#### 반복되는 악순환

1. **신규 담당자 온보딩에 수개월** — 어떤 서버가 있고, 어떤 서비스가 돌아가는지 파악만 하다 시간이 지남
2. **동일 장애 반복** — 이전에 같은 문제가 있었고 해결책이 있었지만, 기록이 없어 다시 처음부터 원인 파악
3. **설정 이유 불명** — "이 방화벽 룰은 왜 있지?", "이 포트는 왜 열려 있지?" 아무도 모름
4. **구성도 부재** — SI 업체 납품 시 문서가 마지막, 이후 변경 사항 미반영

### 3-3. 툴 파편화 — "관리 툴이 너무 많다"

```
현재 일반적인 운영 환경
─────────────────────────────────────────────────────────
서버 모니터링    →  Zabbix 또는 제조사 제공 툴
네트워크 모니터링 →  MRTG / PRTG (무료 버전)
로그 수집        →  별도 Syslog 서버 (설정만 해두고 잘 안 봄)
자산 관리        →  엑셀 파일 (여러 버전이 공유 폴더에 혼재)
IP 관리          →  엑셀 또는 아무 관리 안 함
SSL/도메인 만료  →  달력 메모 또는 관리 안 함
장애 이력        →  이메일 / 카카오톡 / 아무것도 없음
서비스 체크      →  아무것도 없음 (전화로 민원 접수 후 확인)
─────────────────────────────────────────────────────────
→ 툴마다 별도 로그인, 별도 화면, 서로 연동 없음
→ 장애 발생 시 여러 툴을 동시에 열어놓고 수동으로 상관관계 파악
```

### 3-4. 주요 고충 요약

| 고충 | 발생 빈도 |
|------|----------|
| 담당자 교체 시 운영 이력·노하우 전부 소실 | 이직 시마다 |
| 전체 인프라 현황을 한눈에 볼 수 있는 화면 없음 | 매일 |
| 이전 장애와 동일한 문제인데 기록이 없어 처음부터 원인 파악 | 월 수회 |
| SSL·도메인·유지보수 만료를 놓쳐 서비스 중단 | 연 1~2회 |
| 관리 툴이 너무 많아 어디서 무엇을 봐야 할지 모름 | 매일 |
| 구성도가 없거나 수년 전 납품 당시 문서가 마지막 | 작업마다 |
| 디스크 Full·메모리 부족을 알람 터진 후에야 인지 | 연 수회 |

### 3-5. 기존 해결 시도와 한계

| 시도 | 한계 |
|------|------|
| **상용 NMS** (SolarWinds, PRTG) | 라이선스 수천만 원, 모니터링 외 자산·장애 이력은 별도 필요, 설정 전문가 부재 시 방치 |
| **오픈소스 조합** (Zabbix + Grafana + ELK) | 설치·설정 복잡, 통합 화면 불가, 한국어 미지원 |
| **자체 개발** | 개발자 이직 시 소스·노하우 함께 소실 — 인프라 히스토리 문제와 동일 패턴 반복 |

## 4. Digicap이 해결하는 문제

```
기존 문제                              Digicap 해결책
───────────────────────────────────────────────────────────────
담당자 교체 시 히스토리 전부 소실  →  DB에 모든 이력 축적, 사람이 아닌 시스템이 보관
관리 툴 5~6개 파편화 운영         →  모니터링·자산·알림·장애·구성도 단일 플랫폼 통합
자산 대장 엑셀 수동 관리          →  에이전트가 자동으로 자산 등록·상태 갱신
구성도 없음·불일치               →  LLDP/ARP/WWN 기반 토폴로지 자동 생성·실시간 반영
동일 장애 반복                   →  Incident 타임라인·해결 내용 기록으로 재발 방지
변경 이력 없음                   →  Syslog 기반 설정 변경 자동 감지·기록
SSL·도메인 만료 누락              →  매일 자동 체크 + 이메일 사전 경보
용량 위기 사후 대응               →  AI 기반 소진 예측으로 사전 경보
인프라 전문가 없이 운영           →  자동 수집·자동 알림·AI 분석으로 전문 지식 보완
온보딩 3~6개월 소요              →  의존성 맵·구성도·자산 대장·장애 이력으로 즉시 파악
```

## 5. 핵심 기능

### 5-1. 자산 중심 통합 관리

Digicap의 핵심 설계 원칙은 **"자산(Asset)이 모든 데이터의 중심"**이라는 것입니다.

하나의 자산을 등록하면, 그 자산에 연결된 모니터링 메트릭·네트워크 포트·랙 위치·토폴로지 위치·장애 이력·유지보수 계약·변경 이력이 모두 하나의 ID로 연결됩니다.

```
                    ┌─── 실시간 메트릭 (CPU·메모리·디스크·네트워크)
                    ├─── 랙 위치 (Rack A-01, U35-36)
                    ├─── 토폴로지 위치 (스위치 포트 Gi0/1 연결)
     자산 등록 ────→├─── 유지보수 계약 (벤더·만료일·담당자)
     (Asset)        ├─── 변경 이력 (IP 변경, OS 업데이트, 상태 변경)
                    ├─── 장애 이력 (Incident 타임라인·근본 원인)
                    ├─── 의존성 관계 (이 서버가 중단되면 영향받는 서비스)
                    └─── 라이프사이클 (운영 중 → 폐기 예정 → 폐기 완료)
```

**자산 페이지에서 제공하는 기능:**

- 통계 카드: 전체 자산 수, 온라인/오프라인 현황, 계약 미체결 자산 수 한눈에 파악
- 다중 필터: 타입(서버/스위치/방화벽 등) + 상태(온라인/오프라인) + 라이프사이클(운영/폐기예정/폐기완료/반납) 복합 필터링
- 최근 활동 시간: 각 자산의 마지막 통신 시간을 색상으로 구분 (5분 이내=녹색, 1시간 이내=노란색, 초과=빨간색)
- CSV 내보내기: 현재 필터 결과를 CSV로 즉시 다운로드 (감사·보고용)
- 교차 참조: 자산 행 확장 시 랙 위치, 토폴로지 등록 여부, 에이전트 버전을 즉시 확인
- 자산 등록 출처 표시: 수동 등록 / 에이전트 자동 등록 / 네트워크 탐지 자동 등록을 배지로 구분

### 5-2. 실시간 모니터링 (7개 카테고리)

| 카테고리 | 수집 대상 | 주요 지표 |
|----------|----------|----------|
| **물리 서버** | Linux·Windows 서버 | CPU·메모리·디스크 사용률, 네트워크 I/O, 프로세스, Load Average |
| **가상화** | ESXi·KVM·Hyper-V·Proxmox | 하이퍼바이저별 VM 목록, VM당 CPU·메모리·디스크 |
| **네트워크** | 스위치·라우터·방화벽 | 인터페이스 트래픽, 포트 Up/Down, LLDP 인접 장비, Syslog |
| **보안** | UTM·IPS·WAF | 상태 모니터링, 설정 변경 감지 |
| **스토리지** | NAS·SAN·DAS | 볼륨별 사용량, I/O, 서버 연결 매핑 (마운트 포인트·프로토콜) |
| **BMC / 하드웨어** | iDRAC·iLO·IPMI·XCC | 전력 소비(W), PSU 상태, CPU·인렛·아울렛 온도, 팬 속도, 하드웨어 헬스 |
| **서비스 체크** | HTTP·HTTPS·TCP·DNS | 응답 시간, 상태 코드, 가용성 |

**서버 상세 페이지**: 시간대별 CPU·메모리·디스크 I/O·네트워크 I/O 차트, 마운트 포인트별 디스크 사용량, 설치 소프트웨어 목록, BMC 하드웨어 상태까지 하나의 화면에서 확인

### 5-3. 시스템 구성 자동 현행화

**담당자가 바뀌어도 최신 구성도가 항상 존재합니다.**

#### 토폴로지 자동 생성
- **LLDP 자동 탐지**: 에이전트가 수집한 LLDP 데이터로 스위치↔서버 연결 관계 자동 구성
- **ARP/MAC 자동 탐지**: ARP 캐시·MAC 주소 테이블에서 L2 연결 관계 자동 추론
- **WWN 자동 탐지**: 파이버채널 WWN 데이터로 SAN 토폴로지 자동 구성
- **네트워크 스캔**: SNMP·SSH 기반 네트워크 장비 자동 발견 및 등록
- **Physical / SAN 레이어 전환**: 물리 네트워크와 SAN 패브릭 구성을 각각 시각화
- **자동 배치**: Dagre 알고리즘 기반 노드 자동 레이아웃, 수동 배치 후 위치 저장
- **노드 클릭 메트릭 팝오버**: 토폴로지 맵에서 장비를 클릭하면 CPU·메모리·디스크 실시간 메트릭을 즉시 확인

#### 랙 실장 현황
- 데이터센터 랙 단위 장비 배치 시각화 (U 단위 스텐실 아이콘)
- 자산 우선 선택 방식: 자산을 먼저 선택하면 장비 유형에 맞는 스텐실이 자동 적용
- 빈 슬롯·사용 중 슬롯·전체 용량 통계

#### 의존성 맵
- 자산 간 연결 관계를 방향 그래프로 시각화
- "이 서버가 다운되면 어떤 서비스에 영향이 가는가?" 즉시 파악

### 5-4. 자산 관리 통합 (6개 영역)

| 영역 | 기능 |
|------|------|
| **SW 인벤토리** | 소프트웨어 라이선스 추적, 벤더·계약유형·만료일·라이선스 수, 서버별 설치 현황 |
| **IP 관리 (IPAM)** | 서브넷별 CIDR·할당률, IP 주소별 상태(사용/예약/가용), 서브넷 추가·IP 할당 CRUD |
| **도메인 / SSL 인증서** | 만료일 자동 추적, 만료 상태 배지 (만료됨/30일/90일/유효), 자동 갱신 플래그 |
| **변경 이력** | 자산별 필드 변경 자동 기록 (상태·IP·OS·위치·라이프사이클 등), 타임라인 형태 조회 |
| **용량 계획** | 디스크·스토리지·CPU·메모리 트렌드 차트, 임계치 알림, AI 기반 소진 예측 분석 |
| **유지보수 계약** | 벤더·계약유형·기간·담당자 정보, 만료 30일 이내 사전 경보, 대시보드 연동 |

### 5-5. 알림 / 장애 대응

#### 알림 체계
- **알림 규칙 엔진**: 메트릭별 임계치 설정 (CPU > 90% 5분 지속 → critical), 자산별 또는 타입별 적용
- **알림 현황**: 실시간 알림 피드, 심각도·상태별 필터, 일괄 확인·해소 처리
- **이메일 알림**: 알림 발생 시 지정된 담당자에게 이메일 자동 발송

#### 장애 관리 (Incident)
- 장애 생성 → 담당자 배정 → 타임라인 기록 → 근본 원인 분석 → 해결 내용 기록
- **타임라인**: 장애 발생부터 해결까지 모든 이벤트를 시간순으로 기록
- **재발 방지**: 과거 동일 장애를 검색하여 이전 해결 방법 즉시 참조

### 5-6. AI 분석

| 기능 | 설명 |
|------|------|
| **이상 탐지** | LLM이 30일 메트릭 트렌드를 분석하여 이상 패턴 자동 탐지 (위험도·신뢰도·요약·권장사항) |
| **로그 요약** | Syslog 수집 내용을 AI가 패턴별로 분류·요약, 발생 빈도와 권장 조치 제시 |
| **AI 대화** | 자연어로 인프라 상태 질의 ("메모리 사용률이 가장 높은 서버는?", "최근 장애 요약해줘") |
| **용량 예측** | CPU·메모리·디스크 사용량 추세를 AI가 분석하여 소진 시점 예측 및 증설 권고 |

**지원 LLM**: Ollama (로컬, 비용 없음, 인터넷 불필요) / OpenAI / Anthropic — 설정 페이지에서 전환 가능

### 5-7. 리포트

- 기간별 (7/30/90일) 인프라 종합 보고서 생성
- 자산 현황 요약, 알림 통계, 디스크 사용량 TOP, 트렌드 차트
- AI 기반 서술형 분석 포함

### 5-8. 대시보드 — 전체 현황 한눈에

- **장비 카테고리 카드**: 물리서버·VM·네트워크·보안·스토리지·백업별 자산 수, 기동/중지 비율 → 클릭 시 자산 페이지로 이동하여 해당 타입 필터 자동 적용
- **CPU / 메모리 TOP 10**: 가장 부하가 높은 서버 순위 → 클릭 시 서버 상세 페이지로 이동
- **최근 알림**: 실시간 알림 피드 → 클릭 시 알림 페이지로 이동
- **스토리지 용량**: 장비별 사용률 시각화
- **유지보수 만료 임박**: 30일 이내 만료 예정 계약 목록

## 6. 운영 워크플로우 — 하나의 프로세스

Digicap은 인프라 운영의 전체 생애주기를 하나의 연속된 프로세스로 통합합니다.

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        Digicap 통합 운영 프로세스                         │
│                                                                         │
│  ① 자산 등록          ② 자동 구성          ③ 실시간 운영                  │
│  ┌──────────┐       ┌──────────────┐     ┌──────────────────┐          │
│  │ 수동 등록 │       │ 에이전트 설치 │     │ 대시보드 모니터링 │          │
│  │ 또는     │──────→│ 자동 상태 수집│────→│ 알림 규칙 동작    │          │
│  │ 자동 탐지 │       │ 토폴로지 생성 │     │ 서비스 체크       │          │
│  └──────────┘       │ 랙 배치      │     └────────┬─────────┘          │
│                     └──────────────┘              │                     │
│                                                   │ 장애 발생           │
│  ⑥ 폐기/반납         ⑤ 이력 축적           ④ 장애 대응                │
│  ┌──────────┐       ┌──────────────┐     ┌──────────────────┐          │
│  │ 라이프사이클│      │ 변경 이력    │     │ Incident 생성    │          │
│  │ 상태 변경 │◀─────│ 장애 이력    │◀────│ 담당자 배정      │          │
│  │ 폐기 사유 │       │ 알림 이력    │     │ 타임라인 기록    │          │
│  │ 기록     │       │ 용량 트렌드  │     │ 근본 원인 분석   │          │
│  └──────────┘       └──────────────┘     └──────────────────┘          │
│                                                                         │
│  ✓ 담당자 변경 시: 위 모든 데이터가 시스템에 보존                          │
│  ✓ 신규 담당자: 자산 대장 + 구성도 + 장애 이력으로 즉시 업무 파악           │
└─────────────────────────────────────────────────────────────────────────┘
```

### 담당자 변경 시나리오

| 단계 | 기존 방식 | Digicap 도입 후 |
|------|----------|---------------|
| **인프라 파악** | 전임 담당자 메모 찾기, 서버실 직접 확인 (수주~수월) | 대시보드 + 자산 목록 + 토폴로지 확인 (수시간) |
| **구성 이해** | 구성도 없거나 불일치, 직접 trace route 수행 | 자동 생성 토폴로지 + 의존성 맵 즉시 확인 |
| **장애 대응** | 이전 장애 기록 없음, 처음부터 원인 파악 | 장애 이력 검색, 동일 유형 이전 해결 방법 즉시 참조 |
| **유지보수** | 업체 연락처·계약 만료일 모름 | 자산별 유지보수 계약·담당자 정보 즉시 확인 |
| **변경 추적** | "이 설정은 왜 바꿨지?" 알 수 없음 | 변경 이력에서 언제·무엇이·어떻게 바뀌었는지 확인 |

## 7. 도입 효과

### 7-1. 운영 효율화

| 기존 방식 | Digicap 도입 후 |
|----------|--------------|
| 여러 툴 분산 사용 (Zabbix, 엑셀, Syslog 서버 등) | 단일 플랫폼에서 모니터링·자산·알림·장애 통합 관리 |
| 수동으로 장비 대장 업데이트 | 에이전트가 자동으로 자산 등록 및 상태 갱신 |
| 담당자가 직접 서버 접속하여 상태 확인 | 대시보드에서 전체 인프라 현황 즉시 파악 |
| 장애 발생 후 원인 파악에 시간 소요 | 변경 이력·타임라인·의존성 맵으로 빠른 근본 원인 추적 |
| SSL·도메인 만료를 놓쳐 서비스 중단 | 매일 자동 체크 + 이메일 사전 경보 |
| 용량 부족으로 인한 갑작스러운 장애 | AI 기반 용량 소진 예측으로 선제 대응 |
| 구성도가 없거나 수년 전 문서가 마지막 | 토폴로지 자동 생성 + 실시간 업데이트 |

### 7-2. 비용 절감

- **라이선스 비용**: 고가 상용 NMS 대비 자체 구축으로 비용 최소화
- **운영 인력**: 자동화된 수집·알림·집계로 수작업 반복 업무 대폭 감소
- **장애 대응 시간(MTTR)**: 실시간 알림 + 의존성 맵으로 단축
- **예방적 유지보수**: 용량 예측·만료 사전 경보로 긴급 장애 예방

### 7-3. 보안 및 컴플라이언스

- **변경 이력 자동 기록**: 누가 언제 무엇을 변경했는지 자동 기록 — 감사 추적 용이
- **내부망 전용 운영**: 인터넷 연결 없이 온프레미스 환경에서 완전 폐쇄 운영 가능
- **LLM 로컬 실행**: Ollama 사용 시 AI 분석 데이터가 외부로 전송되지 않음
- **사용자 권한 관리**: 관리자/운영자/읽기전용 역할 기반 접근 제어

### 7-4. 타 솔루션 대비 비교

### 자동화 스케줄러

| 주기 | 작업 | 설명 |
|------|------|------|
| **1분** | 알림 평가 | 메트릭 임계치 초과 여부 검사, 알림 자동 생성 |
| **1분** | 오프라인 체크 | 메트릭 미수신 자산 자동 오프라인 전환 |
| **1분** | 서비스 체크 | HTTP·TCP·ICMP 응답 상태 확인 |
| **1분** | Agent 폴링 | Pull 방식 에이전트에서 메트릭 수집 |
| **5분** | 메트릭 집계 | raw 메트릭 → 시간별 집계 테이블 |
| **5분** | BMC 수집 | IPMI·Redfish 센서 데이터 수집 |
| **5분** | AI 예측 | CPU·메모리·디스크 이상 징후 분석 |
| **매일 02:00** | SSL 체크 | 인증서 만료일 재확인 및 알림 |
| **매일 03:00** | 유지보수 체크 | 유지보수 만료 임박 알림 생성 |

### 에이전트 수집 구조

**Push 방식** (에이전트 → 서버): 자산에 경량 에이전트 설치, 1분 주기로 메트릭·Syslog 전송
**Pull 방식** (서버 → 자산): 에이전트 설치 불가 장비(네트워크·BMC)는 SNMP·SSH·Redfish로 서버가 폴링
**Syslog 수신**: 네트워크 장비의 설정 변경·포트 상태 변경·LLDP 이벤트를 자동 파싱하여 이력 기록 + 토폴로지 업데이트
